The wire protocol

The wire format. Types, EIP-712 schemas, constants, frozen at P1.

What it is

The packages/protocol/ package is a tiny TypeScript library with no runtime logic. It exports type definitions, EIP-712 schemas, network constants, and HTLC root helpers. Every other package in the monorepo imports it; the on-chain Solidity in contracts mirrors the same shapes manually so that the typed-data digests on both sides of the wire match byte-for-byte. If you ever wonder where the canonical definition of "what is a channel state" lives in tainnel, this is it.

Why it exists

State channel safety hinges on one promise: the digest a counterparty signs off-chain is the exact digest the contract recovers on-chain when something goes wrong. Drift the field order, swap a uint64 for a uint256, or rename a struct, and signatures stop verifying — usually quietly, until the moment you actually need a dispute to resolve. Centralizing the wire format in a single zero-logic package is how we keep that promise. The state machine, SDK, hub, watchtower, and CLI all read the same constants from the same file. Anyone touching the wire format touches one place, sees one diff, and bumps one version.

Three goals shape the package: (1) zero runtime logic — pure types and constants, no execution paths to test or break; (2) browser-safe — no Node-only dependencies, so the same module ships into a future wallet UI as cleanly as into a server; (3) single source of truth — every other package imports from here, and the Solidity in Adjudicator.sol is treated as a translation of the same definitions.

Key concepts

From the glossary: state, HTLC, cooperative close, EIP-712. Two further terms specific to this page:

  • EIP-712 typed data — Ethereum's standard for signing structured messages. A domain (name, version, chainId, verifyingContract) plus a typed struct produces a deterministic 32-byte digest. Both signer and verifier independently compute the same digest from the same inputs; signatures recover to the original signer's address only if every byte of every field matches.
  • HTLC root — a single bytes32 committing to the set of currently-locked HTLCs in a channel. Computed as a sorted-keccak Merkle root over each HTLC's individual hash. The contract only sees the root inside a state hash; the individual HTLCs live off-chain.

How it works

The package is the bullseye in a small dependency target. Solidity, TypeScript, server, client — they all bend back to the same definitions here.

Mechanically the package is small: five TypeScript files. types.ts defines ChannelState, Update, Htlc, CooperativeClose, and their signed wrappers. eip712.ts declares the typed-data domain and per-struct field arrays in the exact order the Adjudicator hashes them. constants.ts exposes chain id, contract addresses, and value bounds. htlc-root.ts implements the sorted-keccak Merkle aggregation used to commit a set of in-flight HTLCs to a single bytes32. events.ts reserves Nostr event-kind numbers for the future Phase 2 DVM payment-discovery layer (defined here so the allocation is fixed in v1 wire format and cannot drift later).

None of these files do anything at import time. The package is a contract — in the legal sense — between every other package in the repo. Importing a constant from @tainnel/protocol is the project-wide signal that you are agreeing to the v1 wire format and will not silently substitute a different shape.

EIP-712 schemas

The four typed-data structs every signer and verifier shares:

StructFields
ChannelState channelId(bytes32), version(uint64), balanceA(uint256), balanceB(uint256), htlcsRoot(bytes32), finalized(bool)
Htlc id(bytes32), amount(uint256), paymentHash(bytes32), expiry(uint64), direction(uint8) — 0 means A→B, 1 means B→A
Update fromVersion(uint64), toState(ChannelState), expiry(uint64)
CooperativeClose channelId(bytes32), version(uint64), finalBalanceA(uint256), finalBalanceB(uint256), token(address)

The EIP-712 domain is fixed: name="tainnel", version="1", chainId=167000, verifyingContract = the Adjudicator proxy address. Signing with any other domain produces a digest the on-chain Adjudicator will not accept, by construction.

Two design decisions encoded as constants here, both locked in P1:

  • D1.2 — HTLC hash function = sha256. The preimage check in HTLC.verifyPreimage uses sha256 (not keccak256), matching Lightning convention so payment hashes can be reused across systems if desired.
  • D1.3 — HTLC root = sorted-keccak Merkle. Pairs of leaves are sorted before hashing, eliminating proof-order ambiguity. The TS implementation in htlc-root.ts and the Solidity in HTLC.sol agree on this construction.

Tests

One test file: packages/protocol/src/index.test.ts. Because the package is pure types and constants, the only test is a smoke check that all expected exports compile and resolve. There is no logic to unit-test here; the actual signing and verification correctness is covered by the state-machine tests on the off-chain side and the contracts tests on the on-chain side. Run with pnpm --filter @tainnel/protocol test (vitest).

The fact that this package has nothing to mock or stub is deliberate. Logic-free packages do not regress; only their consumers do. If a contract field is renamed or reordered, the type checker in every consuming package fails immediately.

Design constraints

The protocol package is the v1 reference for names, fields, constants, and EIP-712 schemas. Future changes require a wire-format version bump because the EIP-712 domain version is part of every digest the chain verifies — bumping "1" to "2" invalidates every previously-signed message under the old domain. That is the intended migration path.

Deployment facts, such as the addresses in CONTRACT_ADDRESSES[167000], are deliberately separated from protocol shape. A consumer can rely on the type and schema definitions while operational questions about a specific deployment stay with the roadmap and runbooks.
