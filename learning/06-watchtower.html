The Watchtower

The bodyguard. Catches stale-state submissions and slashes the cheater.

What it is

The watchtower is a small Node service whose only job is to sit between the chain and the channel database, watch for unilateral closes, and submit a penalty transaction if the close uses an obsolete state. It lives at apps/watchtower/ and consists of focused modules: watcher.ts for the high-level loop, detector.ts for "is this close stale and is it worth penalizing", responder.ts for building and submitting penalty calldata, storage.ts for encrypted state-backup persistence, plus config.ts and logger.ts.

Architecturally the watchtower is the simplest of the long-running components. It does not hold funds, it does not forward payments, it does not maintain channels. It does one thing: when it sees a closure on chain, it reads the channel's signed-state log from disk, picks the freshest state it has, and races to submit it before the dispute window closes.

Why it exists

The whole point of the dispute window in PaymentChannel.sol is to give honest parties a chance to overrule a stale unilateral close. But "honest party present and online" is not something a payments network can take for granted. Real users are sometimes asleep, real laptops sometimes die, real CLIs sometimes get killed by an OS update. Worse, the dispute window is finite — by default 24 hours — and missing the window means the cheater's stale state stands and the cheater walks off with money they didn't earn.

The watchtower exists so that the user does not have to be online for the system to remain safe. It is, in effect, a stand-in observer that wakes up on every relevant chain event and answers a single question: "is this the most recent state we know of, or do I have something newer signed by both sides?". If newer, it slashes.

Trust model: a watchtower cannot steal funds (it never custodies any), but it can fail to post a penalty in time, either by going offline itself or by being the wrong infrastructure on the wrong network at the wrong moment. The mitigation is redundancy. The hub runs one. Sophisticated users may run their own. Nothing in the protocol depends on a single watchtower being honest.

Key concepts

Decisions D6.1 through D6.4

  • D6.1 Self-hosted only in v1. A "watchtower-as-a-service" mode where third parties run watchtowers on others' behalf is Phase 2. v1 watchtowers protect the operator that runs them.
  • D6.2 Penalty trigger at 50% of the dispute window. With a 24-hour window, the watchtower acts at T = 12 hours — leaving a 12-hour buffer for chain congestion, gas-price spikes, RPC outages, and reorgs. Acting earlier risks acting on a still-finalizing reorg; acting later risks missing the deadline.
  • D6.3 Shared SQLite database. The watchtower reads the same database file the SDK writes to. They run on the same machine. This is a drastic simplification: no replication, no message bus, no extra protocol — the SDK signs a state, writes it to disk, and the watchtower sees it on its next poll.
  • D6.4 Hub-side watchtower covers all hub channels; users may additionally run their own. The hub-side watchtower MUST be hosted on different infrastructure from the hub itself — different region, ideally different cloud provider — so a single outage cannot take down both.

Pairing with tainnel listen

An agent that runs tainnel listen as a long-lived process MAY subscribe to the same chain events that the watchtower watches; the SDK gives it everything it needs to do so. But the watchtower remains the canonical penalty submitter, because users should not need to keep their CLI alive 24/7 just to stay safe. If a user is offline during fraud, recovery is straightforward: they restart listen-mode, the SDK reads chain history, sees that the watchtower already penalized the cheater and that the channel was finalized in their favor, and proceeds. There is no remediation step left to perform.

Functions to call on chain

  • submitPenaltyProof(channelId, penaltyState, signature) — the canonical watchtower call. Slashes 100% of the channel deposit if the proof is valid (state signed by the closer with a version higher than the one the closer posted).
  • dispute(channelId, state, sigCloser) — alternative path: submit a fresher co-signed state without invoking the penalty. Less common; submitPenaltyProof is preferred because it strictly dominates from the honest party's perspective.
  • finalize(channelId) — call once the dispute deadline passes to release funds to whichever side the contract has decided is honest.

Events to monitor

Four event types from PaymentChannel.sol: ChannelOpened (so the watchtower knows about new channels worth watching), ChannelClosingUnilateral (the trigger), DisputeRaised (someone else got there first), and ChannelFinalized (we can stop watching this one).

How it works

Concretely: the hub goes evil and posts a stale state at T = 0. The chain emits ChannelClosingUnilateral with the posted version. The watchtower sees the event on its next poll, checks the version against what it has on disk, and if its disk has something fresher it queues a penalty submission. At T = 12h, half the window, the responder builds the calldata, signs with the watchtower's private key, estimates gas, submits, and retries with bumped gas every 60 seconds if the transaction does not mine. By T = 24h the dispute window has closed; whether the watchtower or the user themselves calls finalize next is irrelevant — the contract knows the channel was penalized, and pays out 100% of the pot to the honest party.

The contract logic is mechanical and not specific to the watchtower: the closer (the hub) signed a state at version 2; the watchtower presents another state at version 5 also signed by the closer. Five is greater than two, the signature recovers to the closer's address, the EIP-712 domain matches, and so the penalty is recorded. Any fresher state would do; the watchtower happens to have the freshest one because it has been writing every signed state to its shared SQLite as it goes.

Verification model

The watchtower is verified in two layers. Unit tests cover stale-version detection: given a latest known signed state and an observed unilateral close version, the detector decides whether a penalty is needed. Integration scenarios should add the chain: open a channel on an anvil fork, submit an older state, let the watchtower observe it, submit the newer evidence, and finalize with the honest party paid correctly. A recovery scenario also matters: the user is offline during fraud, the watchtower handles it, and the user's local state catches up later from chain history.

Design constraints

The watchtower is deliberately smaller than the hub. It does not route payments or invent state; it only remembers signed states, watches chain events, and submits evidence when an on-chain close is older than the latest state it has seen.

  • Latest-state only: the watchtower submits the newest signed state it has observed; it must not synthesize or sign a replacement state.
  • Independent failure domain: it runs separately from the hub so a hub outage or compromise does not also silence dispute response.
  • Idempotent response: duplicate events and retries should not create conflicting transactions or repeated bookkeeping.
  • Reorg tolerance: event handling waits for confirmations and can replay recent history after an RPC disconnect.
  • Observable failure: missed RPC, late evidence, and failed penalty submissions must be logged and exported as metrics because silence is dangerous.
