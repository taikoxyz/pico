Agent runtime

The CLI. The v1 surface every AI agent reaches for first.

What is it

The tainnel CLI in apps/cli is the canonical interface for v1. It bundles every operation an agent (or its operator) needs into one binary: generate an encrypted hot key, open and close channels, send payments, and run a long-lived listen mode that accepts incoming HTLCs. There is no separate daemon, no MCP server, no React wallet UI in v1. An any-language agent shells out to tainnel pay --json and parses one JSON object per state transition; a receiver runs tainnel listen in the background.

Why it exists

State channels structurally need a process that holds a hot key, signs frequent off-chain messages, and (for receivers) keeps a connection open to the hub so it can accept inbound HTLCs and reveal preimages. The simplest design that satisfies all of that without locking the project into one ecosystem is a CLI with a listen subcommand — the same architecture Lightning's lnd + lncli uses, just slimmed to one binary. CLI-as-agent-surface gives every language ubiquitous access (any agent framework can shell out), avoids coupling to MCP or x402, and keeps the trust boundary clean: the key never leaves the local process.

The React wallet UI was deleted from the tree as part of the v1 re-scope. Its outline is preserved in docs/plans/07-agent-runtime.md for a future Phase 2 starting point. v1 is agents only.

Key concepts

  • Signer interface — the SDK abstraction (defined in P4) over key custody. v1 ships exactly one backend, lives in this CLI: encrypted hot key file. Future backends — Nitro Enclave, Turnkey, KMS, EIP-7702 delegation — implement the same interface.
  • Encrypted hot key file — passphrase-derived key (scrypt, N=2¹⁷, r=8, p=1) seals the raw private key with XSalsa20-Poly1305 (libsodium crypto_secretbox). Default path $XDG_CONFIG_HOME/tainnel/key.enc, permissions 0600 enforced at write.
  • tainnel listen mode — long-running daemon; holds a WebSocket session to the hub, accepts inbound HTLCs, reveals preimages, optionally watches chain events. Same binary as the rest of the CLI; just a different subcommand.
  • JSON output — every transitional command supports --json, printing one JSON object per state transition on its own line. Lets a Python or Rust agent parse without writing a viem stack.

How it works

The sender path is straightforward: tainnel pay spawns, prompts for the passphrase (or reads it from TAINNEL_PASSPHRASE in CI), unlocks the encrypted key in memory, builds a ChannelClient from the SDK, calls client.pay(...), prints one JSON line per state transition, and exits when the payment settles.

The receiver path is the long-lived branch and is where the safety-critical logic lives. tainnel listen decrypts the key once at startup, connects the WebSocket to the hub, and enters a loop. For each inbound HTLC offer the hub forwards: validate with the state-machine, generate (if originating) or reveal (if forwarding) the preimage, sign the counter-state, persist before sending the ack, then ack. On disconnect/reconnect, a per-channel resume cursor in the SDK storage makes the loop idempotent — replays do not double-spend. SIGINT triggers a graceful drain: finish the in-flight HTLC if possible, persist the journal, close the WebSocket, exit zero.

The single most safety-critical line in the agent runtime is the ordering between "persist counter-state to storage" and "ack to hub." Persist first, ack second. If you ack first and crash before persist, your local state lags one version behind what the hub has — and what an evil hub could later post on chain. Reviewer attention focuses on this in P8.

Quickstart

pnpm install
pnpm tainnel keys init
pnpm tainnel channel open --hub https://hub.example.com --amount 25

# pay (sender, invoice mode)
pnpm tainnel pay --invoice "$INVOICE" --json

# pay (sender, keysend mode)
pnpm tainnel pay --to 0xRecipient --amount 0.05 --keysend --recipient-pubkey 0x... --json

# receive (run in the background)
pnpm tainnel listen --hub https://hub.example.com &

Verification model

Unit tests cover command parsing, output formats, key-file behavior, signer resolution, config paths, invoice envelopes, hub status, and error paths. The integration test wires two CLI-facing clients through the mock hub: Bob creates an invoice and listens, Alice pays it, and the SDK settles the HTLC using the same message shapes the real hub consumes. Run with pnpm --filter @tainnel/cli test.

Design constraints

  • CLI as agent surface: any-language agents can shell out to tainnel pay --json instead of linking a TypeScript SDK.
  • One binary, two modes: one-shot commands send payments and manage channels; tainnel listen is the long-running receiver process.
  • Passphrase key files: the default signer backend keeps a local encrypted key on disk; raw private keys are for test/CI paths.
  • Machine-readable output: JSON mode emits parseable transition records so non-TS agents can react to settlement, receipts, and errors.
  • Dev loops stay local: dev mock-hub and dev anvil-fork give agent authors a local path before touching live services.
