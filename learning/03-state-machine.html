State machine

Pure functions. The off-chain rules every channel obeys.

What it is

The state machine is the package packages/state-machine. It is a tiny library of pure, deterministic, I/O-free TypeScript functions that describe how a channel evolves: how a new state is validated against the previous one, how an HTLC is added or settled or expired, how a typed-data digest is built from a state so it can be signed. There is no network here, no disk, no chain, no clock outside what callers pass in. Hand the same inputs to the same function on a phone, on a hub, in a watchtower job, or inside a fuzz test, and you get the same bytes back. That property is the entire reason this package exists.

Why it exists

A state channel works because two parties keep agreeing, off-chain, on a single ledger of who owns what. The on-chain contract ultimately enforces that ledger, but only when something goes wrong. Between disputes, the rules of motion live entirely off-chain. If the SDK and the hub disagreed by even one bit on what a valid update looks like — or on the digest a counterparty signed — payments would either get stuck or, worse, slip through unsigned. This package is the single source of truth that every off-chain actor must agree with.

It is also the oracle for the on-chain side. Every hash that the contract computes in Solidity must match, byte for byte, what this TypeScript code computes in Node or in a browser. Otherwise a state signed off-chain will fail to verify on-chain at exactly the moment a victim needs it most: during a dispute. That cross-package equivalence is so important the docs treat it as an explicit acceptance criterion.

Mental model: the contract is the on-chain mirror of this package. Same inputs, same outputs, same digests. Two implementations of one specification — one in Solidity for the chain, one in TypeScript for everywhere else.

Key concepts

A ChannelState is a snapshot: (channelId, version, balanceA, balanceB, htlcs[], finalized). Versions only ever go up. An Update carries fromVersion, toVersion, and a nextState — never a delta. The library's job is to say yes or no to a transition, then return the new state if yes. An HTLC is a conditional payment held inside the state: amount, paymentHash, expiry, direction. While an HTLC is pending, its amount is debited from the sender's balance but not yet credited to the receiver — the funds live in the HTLC slot. Settling reveals a preimage and credits the receiver. Failing returns the funds to the sender. Expiring is the same as failing, just driven by time.

The htlcsRoot is a single bytes32 commitment to the entire pending-HTLC set, computed by sorting the HTLCs by id, abi-encoding each, concatenating, and taking a single keccak. It is what gets folded into the EIP-712 digest of a state, so a signature over a state implicitly commits to the exact HTLC set. Sorted-keccak (rather than a binary Merkle tree) keeps the root cheap to compute on-chain — the contract has at most five HTLCs to handle in v1.

How it works

The flow is short and rectangular. Given a current state at version N and an incoming update, validateUpdate checks four things in order: the channel id matches, the channel is not already finalized, the new version is strictly greater than N (via ensureMonotonicVersion, which throws StaleVersionError), and the total balance — on-channel balances plus the sum of pending HTLC amounts — is preserved. If any check fails, an error is thrown and nothing else happens. If all four pass, applyUpdate simply returns the proposed nextState. The function is pure: no mutation, no side effects.

HTLC operations sit on top of the same pattern. addHtlc debits the sender's balance and pushes an entry into the array, refusing duplicate ids and insufficient balances. settleHtlc finds the entry, removes it, and credits the receiver. failHtlc removes it and refunds the sender. expireHtlcs(state, now) walks the list and fails everything past its expiry. None of these functions mutate; each returns a new state. The signing.ts module builds EIP-712 typed-data structures (buildChannelStateTypedData, buildUpdateTypedData, buildHtlcTypedData, buildCooperativeCloseTypedData) so that whichever Signer the caller plugs in produces a digest the contract will accept.

What's not here yet: signing.ts currently only builds typed data. The hashChannelState, verifyChannelStateSignature, and matching helpers for Update/Htlc/CooperativeClose are not implemented. computeHtlcsRoot exists upstream in @tainnel/protocol as htlcMerkleRoot; the spec-compliant sorted-keccak path still needs cross-validation against the contract.

Gaps vs production-ready

Sourced from docs/plans/03-state-machine.md. This is the v1 critical path: P4 SDK, P5 Hub, P6 Watchtower, and P7 Agent runtime are all blocked on these landing.

  • Implement hashChannelState, hashUpdate, hashHtlc, hashCooperativeClose using viem's hashTypedData.
  • Implement verifyChannelStateSignature, verifyUpdateSignature, verifyCooperativeCloseSignature via viem's recoverTypedDataAddress.
  • Implement computeHtlcsRoot matching D1.3 exactly: sort by id (lexicographic on bytes32), abi-encode each as (id, amount, paymentHash, expiry, direction), concat, single keccak256.
  • Implement verifyPreimage using sha256 (D1.2), tested against contract output.
  • Generate packages/state-machine/test/fixtures/oracle.json with ~50 random inputs and their expected hashes/roots; have forge fuzz tests cross-validate via vm.readJson.
  • Property tests with fast-check: monotonic version, balance conservation, idempotent applyUpdate, addHtlc → settleHtlc round-trip, addHtlc → failHtlc round-trip, replay rejection, sorted-keccak shuffle invariance.
  • Strict invariants: reject amount == 0 HTLCs, reject duplicate HTLC ids, reject finalized: false → true while HTLCs are pending.
  • Coverage gate: pnpm --filter @tainnel/state-machine test --coverage ≥ 95% lines and branches, wired into CI.
