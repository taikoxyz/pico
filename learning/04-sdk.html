The SDK

The library every agent calls. ChannelClient, Signer, transport, storage.

What it is

The SDK is the package packages/sdk. It is the concrete library that agent authors actually instantiate when they want to open a channel, send a payment, or accept inbound HTLCs. Everything below it — the wire format in protocol, the rule book in state-machine — is pure data and pure functions. The SDK is the first layer that touches the outside world: it opens WebSockets, writes files, generates randomness, and orchestrates the back-and-forth that turns an intent like "pay Bob 0.05 USDC" into a sequence of signed messages and on-disk records.

The library is built around one class, ChannelClient, plus three pluggable interfaces it composes: a Signer for key custody, a Transport for hub messaging, and a ChannelStorage for durable per-channel state. Everything is dependency-injected. There is no global, no implicit network, no implicit disk; the same client construction works in a Node script, in a test harness, and (eventually) in a browser.

Why it exists

The state-machine package can tell you whether a state transition is valid, but it cannot send the corresponding signature anywhere. Someone has to actually do the network calls, hold the latest signed state on disk, persist before sending so a crash mid-flight does not strand a payment, and reconnect with state replay when a hub session drops. That orchestration logic is the SDK's job, and it is the same regardless of whether the consumer is a CLI, a long-running daemon, or a future wallet UI. Putting it in one place keeps the surface that agent authors learn small, and keeps the safety-critical persist-before-send rule in exactly one function.

Audience for v1: agent authors writing a Node/TS script, or the apps/cli which uses this SDK as the canonical agent runtime. The wallet-UI consumer is deferred to Phase 2; the quickstart docs target a CLI / agent-script audience, not a browser.

Key concepts

A Signer abstracts every EIP-712 signature the channel needs. The interface is intentionally small — address(), signChannelState, signUpdate, signCooperativeClose, signHtlc — and ECDSA-output. Hashing for those typed-data structs lives in the state-machine package; the Signer is purely a key-custody adapter. The v1 backend is an encrypted hot key file (passphrase-derived scrypt key, sealed with XSalsa20-Poly1305, permissions 0600, implementation lives in apps/cli). Future backends — KMS, Turnkey, Nitro Enclave, EIP-7702 delegation — implement the same interface; the SDK never calls a private key directly.

A Transport is a request/response message pipe. The v1 implementation is WebSocketTransport: every message is JSON {id, kind, payload}, the transport keeps a Map<id, deferred> for request/response pairs, heartbeats every 30 seconds, and reconnects with exponential backoff (200ms → 30s, infinite, jittered). On reconnect it pushes the latest signed state per channel so the hub can replay any in-flight HTLC. NostrRelayTransport is a Phase-2 stub that compiles but throws.

A ChannelStorage persists channels and their latest signed states. MemoryStorage is for tests. IndexedDBStorage is the browser default. FileStorage is the Node default and uses the atomic-rename pattern (write tmp → rename) with one JSON file per channel under ${root}/channels/${id}.json. The decision matrix (D4.1) is: browser → IndexedDB, Node → file.

How it works

The architecture is layered. At the bottom sit the pure layers: state-machine and protocol. The SDK wraps three pluggable concerns around them. ChannelClient is the orchestration glue on top. Above the client sit the actual consumers — the CLI's tainnel pay and tainnel listen subcommands, and (on the operator side) the hub. Most agent authors never instantiate ChannelClient directly; they shell out to the CLI, which does so for them. But the surface is exactly the same in either case.

The flow inside pay() is the most safety-critical sequence in the SDK. The client generates a 32-byte preimage, hashes it, builds an Htlc, applies it locally with state-machine.addHtlc, and asks the Signer to sign the resulting state. The next step is the rule that anchors everything else: persist before send. The new signed state is written to disk before the signature ever leaves the process. If the SDK crashes between sign and send, on restart the persisted state is what the SDK believes; if the hub already received the signature, it can re-deliver and the SDK matches it. If the SDK signed but didn't persist and didn't send, the signature is lost and the world is consistent. The dangerous middle case — signed and sent but not persisted — is the one this rule eliminates. After persistence, the client sends the pay message, awaits a payment.settle with the preimage, verifies it matches the paymentHash, applies settleHtlc locally, signs the settled state, and persists again.

Why persist-before-send matters: if the hub gets a signature but the local store doesn't have the corresponding state, the hub can later post that state on-chain and the client cannot challenge with anything newer.

open() and close() follow the same shape. Open submits an openChannel tx via viem, awaits the ChannelOpened event, persists the channel as open, and subscribes to the hub WebSocket. Close has two paths: cooperative (send close.request, await counter-signed close-state, submit closeCooperative), with a 60-second timeout that falls back to closeUnilateral against the latest signed state on disk. The CLI's tainnel listen is a long-running consumer that holds a session open and emits htlc:incoming / htlc:settled events; the SDK must allow constructing a ChannelClient without an open channel so the daemon can subscribe based purely on storage state.

Gaps vs production-ready

Sourced from docs/plans/04-sdk.md. The SDK is not started — interfaces only; every method on ChannelClient, WebSocketTransport, and pay() currently throws not implemented.

  • Define the Signer interface in packages/sdk/src/signer.ts with address(), signChannelState, signUpdate, signCooperativeClose, signHtlc.
  • Provide an InMemorySigner (test-only) re-exported from @tainnel/test-utils, never from the public SDK entry point.
  • Real WebSocketTransport implementation: backoff per D4.2, JSON framing {id, kind, payload}, request/response Map, heartbeat ping every 30 seconds, reconnect after two missed pings, state replay on reconnect.
  • FileStorage with the atomic-rename pattern; IndexedDBStorage as a tiny wrapper (no Dexie, no idb).
  • Real ChannelClient.open / .pay / .close implementations, including the persist-before-send line in pay() and the 60-second cooperative-to-unilateral fallback in close().
  • Persistence-survives-crash test: sign, crash-before-send, reload, assert recovery.
  • Quickstart in packages/sdk/README.md targeting the CLI / agent-script audience (not a browser), runnable against the @tainnel/test-utils mock hub. Cross-link to apps/cli for authors who do not want to write TS.
  • Coverage gate: pnpm --filter @tainnel/sdk test --coverage ≥ 80% lines.
