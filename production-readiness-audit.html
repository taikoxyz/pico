Overall Verdict: NOT READY for Taiko mainnet GA

The protocol design and v2 contracts are sound (181/181 Forge tests pass; reentrancy + CEI verified; no critical contract-level bugs found). However, four classes of issue block mainnet:

  1. One Critical doc-vs-code drift in EIP-712 domain (version: '1' in spec, '2' in code) — external SDK authors mis-sign on first attempt.
  2. Two newly identified High runtime issues: a concurrent-pay race in the hub router (R-01) and a payment durability orphan-state window (R-02). Both are pre-existing in HEAD.

Estimated time to GA: 5–8 weeks of focused work. Critical path: §9 Action Plan items 1–10.

1.2 New Findings (post-baseline)

CRITICAL N-01 · EIP-712 domain version drift between spec and deployed code

docs/protocol-spec.md (§0, §6.1, §7 at lines 478, 484, 534) declares the EIP-712 domain version: '1'. The deployed code uses version: '2' at packages/contracts/src/Adjudicator.sol:118 and packages/protocol/src/eip712.ts:4 (EIP712_DOMAIN_VERSION = '2'). The TS test at eip712.test.ts:44 uses '2'. Any external integrator writing a signer against the published "frozen, normative" spec will produce signatures that do not verify against the on-chain Adjudicator. Single-word doc fix, but every external SDK author hits this on first attempt.

Fix: rewrite §0, §6.1, §6.2, §7 of docs/protocol-spec.md to declare version: '2'; document the v1→v2 typed-data additions (htlcsCount, htlcsTotalLocked, on-chain HTLC settlement).

HIGH N-02 · Spec §6.2 omits htlcsCount and htlcsTotalLocked from ChannelState

docs/protocol-spec.md:500 lists ChannelState fields as channelId, version, balanceA, balanceB, htlcsRoot, finalized. The on-chain typehash (Adjudicator.sol:85-87) includes htlcsCount, htlcsTotalLocked. EIP-712 hashing diverges for any signer following the spec literally.

Fix: §6.2 must list the two additional fields in typehash order.

HIGH N-03 · Spec §5.2 contradicts §5.4 on closeUnilateral and HTLCs

§5.2 (line 348) says state.htlcsRoot MUST be bytes32(0). Code (PaymentChannel.sol:362-390) accepts non-empty roots; §5.4 (v2 on-chain HTLC settlement) supersedes §5.2 but the contradictory line remains.

Fix: remove the stale line from §5.2.

MEDIUM N-04 · closeUnilateral accepts dual-signed state with version == 0

PaymentChannel.sol:362-390 does not require s.version > 0 nor s.version > ch.postedVersion. A buggy hub producing a v=0 dual-signed state could close at zero balances. Defense-in-depth: require(s.version > 0, "v=0");.

MEDIUM N-05 · topUp does not re-check allowedTokens

PaymentChannel.sol:429-498. If the owner de-allowlists a token after a Circle pause incident, existing channels can still pull through safeTransferFrom on topUp. Likely intentional (avoid stranding users) but undocumented. Add to threat model.

LOW N-06 · closeCooperative does not check signedAt ≤ validUntil on-chain

PaymentChannel.sol:336-356. Off-chain SDK enforces it. Both signatures required, so a malicious counterparty cannot exploit a degenerate window unilaterally. Defense-in-depth fix only.

LOW N-07 · Stale line numbers in docs/audit-status.md

PC-01 cites PaymentChannel.sol:255 for dispute(); actual location is :519. Evidence column line numbers across PC-01, PC-04, PC-06 drift from current source.

2.2 Runtime Safety Findings (new)

CRITICAL R-01 · Outgoing-channel race in handlePay

The comment at apps/hub/src/api/ws.ts:255-256 claims "the per-outgoing-channel mutex is taken inside router.route via the channel-pool's existing locks." This is false. router.route (router.ts:83-165) only reads channelPool.latest(), a non-locking getter (channel-pool.ts:100-102). Two concurrent pays routing to the same outgoing channel through different incoming channels can both:

  1. Read the same latestOutgoing version N.
  2. Each sign a distinct version N+1 state with different HTLCs.
  3. First grabs the outgoing mutex, persists v(N+1). Second grabs the mutex, but recordState silently drops because state.version ≤ existing.version (channel-pool.ts:67). The second payment's outgoingHubSigned is in payment_routes but not in signed_states → on restart the hub serves a stale view, and the recipient can present the higher-versioned (signed) state for dispute.

Fix: route under the outgoing-channel lock once the router resolves it (move router.route() inside the existing inner mutex, or split findChannelBetween then take the lock then read+sign).

HIGH R-02 · Hub payment durability — orphan signed-state window

ws.ts:300-342 inserts into htlcs, payments, payment_routes in a single DB transaction. But channelPool.recordState for the outgoing happens outside that transaction (line 297). Crash window: outgoing-channel signed state persists, but htlc/route rows fail → on restart, payment_routes lacks the row, in-memory inflight is empty, hub has signed an outgoing state with no audit trail. Recipient can replay it.

Fix: persist the outgoing state inside the same DB transaction, or sign-and-store before the channel-pool write.

HIGH R-03 · No WS payload size limit or rate limiting

@fastify/websocket does not enforce a frame size by default. A malicious peer can submit a 100 MB JSON frame. grep for @fastify/rate-limit returns 0 hits — anyone with a valid envelope can flood pay messages. Per-channel mutex provides backpressure but the message queue is unbounded.

Fix: register websocket plugin with { options: { maxPayload: 65536 } }; add @fastify/rate-limit with per-IP and per-signer limits; add WS connection cap.

MEDIUM R-04 · Watchtower concurrent submitPenalty nonce collision

watchtower/responder.ts:167 re-reads pending nonce per call. Scheduler iterates synchronously, but index.ts:351-355 calls submitPenalty from an event handler that can fire concurrently if multiple closeUnilateral events arrive in one poll. Two parallel calls pick the same nonce → one reverts.

Fix: add a wallet-level mutex around submitPenalty.

MEDIUM R-05 · Watchtower reorg-during-included-penalty

If a previously-included penalty tx is reorg'd out, getInFlight still says "included" and the watchtower will not re-submit. Hub chain-watcher.ts:276 has rewindForReorg, but the watchtower's storage.ts doesn't track block-hash for observations — only meta key last_processed_block.

Fix: store blockHash per observation; re-evaluate inclusion on rewind.

MEDIUM R-06 · Per-counterparty HTLC cap is hardcoded

router.ts:334-340 hardcodes per-token caps (ZERO=1 ETH, PTST=100 PTST, USDC=MAX_HTLC_VALUE_PER_COUNTERPARTY). Not config-driven; new tokens require code change. Not exposed on /v1/info; clients can't surface "amount exceeds per-counterparty cap" and get a generic router: ... error.

Fix: env-driven config per token; expose on info endpoint.

LOW R-07 · KeyedMutex memory leak

apps/hub/src/mutex.ts:11-22 never deletes entries. Long-running hub processing many ephemeral channels grows the map unboundedly (~64 bytes each). Add delete(key) when lock.tail resolves to Promise.resolve().

LOW R-08 · lockOrder dead code

ws.ts:111-113 defines lockOrder but never calls it. If the intent was canonical-order locking to prevent deadlock, that intent is unimplemented. Either wire it or delete.

LOW R-09 · CLI listen regenerates keysend keypair every run

apps/cli/src/commands/listen.ts:105. A keysend payment in flight at restart cannot be decrypted by the new instance. Fund-safe (HTLC still settles) but memos lost. Fix: persist keypair beside channel storage, or drop keysend from v1.

LOW R-10 · Watchtower has no _migrations table

watchtower/storage.ts:136 uses IF NOT EXISTS only; schema changes must stay backward-compatible or recreate. Adopt hub's _migrations pattern.

7.3 Strategic Insights

Where pico wins

  1. Trustless HTLC settlement. Every x402-style direct-tx competitor settles on-chain per payment → gas cost per tx at scale. Pico amortizes gas across thousands of in-channel updates.
  2. Sub-cent unit economics at sustained rates. 1M-call API session ≈ $200 in pico fees (0.02% × $1M); same on Solana Pay ≈ $5,000; on Base x402 ≈ $100–200 in L1 fees alone over time.
  3. Deterministic 1-hop routing. No facilitator-discovery overhead like x402; no congestion failures like Solana under load.
  4. No facilitator counterparty. x402 requires Coinbase/Stripe/Cloudflare; pico hub is structurally non-custodial (HTLCs).
  5. No protocol token / no governance. Real moat vs SKALE (SKL), Connext (CLEAR), Hyperliquid (HYPE).

Where pico loses

  1. Adoption gap is structural. x402 = 169M+ tx, ~$600M annualized; pico is pre-GA. The narrative race is over — pico must compete on TCO at high volume, not on standard-setting.
  2. No facilitator discovery. x402 is a discovery layer; agents find paid APIs through Coinbase/AWS marketplace. Pico has no equivalent.
  3. Taiko vs Base/Solana/Stellar in TVL & developer mindshare. Base + Solana are where x402 settles. Taiko's stablecoin TVL is materially smaller.
  4. No streaming/session model surfaced. Stripe MPP (Feb 2026) standardised session pre-auth + streamed micro-payments — semantically what pico channels already do, but not marketed that way.
  5. No ERC-8004 trustless-agent identity. SKALE shipped this Jan 2026.
  6. Privacy gap widens. SKALE Private MPPs (Mar 27, 2026) hide on-chain amounts; pico has no privacy story.

Single most pressing threat: x402 (+ its ecosystem)

x402 wraps Coinbase + AWS Bedrock AgentCore + Stripe + Cloudflare into a single de-facto agent-payment standard. It will be the default for AI agent payments in 2026. Pico cannot win the standard-setting race; it must position as the channel-layer backend underneath x402 for high-frequency callers. Concretely: ship an x402-compatible facilitator endpoint on top of pico channels so x402 calls settle through the user's open pico channel when both parties have one. This converts x402 from competitor to distribution.

Recommended pico positioning shifts

  • "Pico is the channel-layer accelerator for x402 / MPP." Frame pico as complementary infrastructure, not a competitor.
  • Document pico channels as MPP-compatible sessions: pre-authorized spend, streamed micro-payments, deterministic settle.
  • Concrete TCO table at scale: 1M API calls = $200 on pico vs $5,000 on Solana Pay vs $200 + $X facilitator-fee on x402.
  • Drop "competing with Lightning" framing. Lightning now has USDT (Taproot Assets); head-to-head on stablecoin liquidity is unwinnable. Pivot to "pico is to Taiko what Lightning was to Bitcoin — but for sustained high-frequency callers on EVM."
