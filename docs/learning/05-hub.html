The Hub

The router. Holds channels with everyone, forwards HTLCs, watches the chain.

What it is

The hub is the always-on intermediary that makes the topology of this network feasible. Every other actor in the system is, in principle, transient: an agent might pay once and never again, a watchtower might be self-hosted and only run when its operator's laptop is awake, a CLI is born and dies inside a single call. The hub is none of those things. It is a Fastify HTTP server with a native WebSocket plugin, fronted by a SQLite database, that stays online so that other actors do not have to. Its source lives at apps/hub/ and its pieces are exactly what you would expect: server.ts for HTTP and WebSocket transport, an api/ directory of endpoint handlers, channel-pool.ts tracking every open channel, chain-watcher.ts reading on-chain events, dispute-handler.ts assembling penalty calldata, fee-policy.ts computing what the hub charges per hop, router.ts deciding whether and how to forward a payment, liquidity.ts tracking inbound and outbound capacity per channel, and a db/ directory with raw SQL migrations and small repos (no ORM).

router.ts is the most subtle file in the hub package because it carries the expiry-buffer arithmetic that determines whether a forwarded HTLC is safe to take on. The surrounding modules split responsibilities cleanly: channel pool for local state, liquidity for reservations, chain watcher for on-chain events, dispute handler for counter-states, and API routes for REST/WebSocket entrypoints.

Why it exists

Lightning's LSP (Liquidity Service Provider) model is the design we are copying, almost gesture for gesture. Without a hub, a network of n agents that all want to be able to pay each other would need on the order of n² channels: every pair would have to fund and rebalance their own escrow. With one hub in the middle, every agent only ever needs one channel — to the hub — and the hub fronts the rest. Any agent in the network can pay any other, in one logical hop, by going through the hub.

The hub is therefore in a competitive market position rather than a privileged one. Its product is liquidity, fees, and uptime. If it goes offline, agents close their channels and find another. If it charges too much, a competing hub takes its volume. The trust model below makes this work: the hub never holds funds it can run away with, only liquidity it can refuse to deploy.

Key concepts

What a hub cannot do

The hub is not a bank. It is closer to a postal service that gets paid per envelope delivered. The contracts and the off-chain state machine pin down its capabilities tightly.

  • It cannot steal funds. Every state update is signed by the user. The hub holding a stack of the user's signed states does not let it move money against the user's will, because moving money requires the user to sign a newer state that authorizes it.
  • It cannot forge balances. The Adjudicator on-chain only accepts states with valid EIP-712 signatures from both parties. The hub cannot synthesize a signature it does not have.
  • It cannot bypass HTLC mechanics. Inbound and outbound HTLCs in a routed payment are linked by the same hash. The only way the hub gets paid upstream is by revealing the preimage downstream. Either both legs settle or both expire.

What a hub can do (failure modes you must plan for)

  • Refuse service. The hub may decline to forward a payment for any reason; the upstream HTLC will then expire harmlessly.
  • Stall. A hub that goes offline mid-payment leaves an HTLC dangling. After the timeout the upstream side reclaims. The user's eventual remedy is unilateral close.
  • Try to publish a stale state. In a unilateral close, the hub might pick a snapshot of the channel where it owned more of the pot. This is the attack the watchtower exists to defend against (see next page).

Decisions D5.1 through D5.4

Four operational decisions shape the hub's shape:

  • D5.1 SQLite + litestream → R2 backup. One hub, one process, one file. Simpler than Postgres, and litestream gives continuous off-site replication.
  • D5.2 Per-message signed envelope auth. The hub does not maintain login sessions. Each WebSocket message carries its own signature and a nonce; this means a reconnect costs nothing.
  • D5.3 Hot wallet for hub on-chain operations, with a balance cap and periodic sweeps to a cold address. The wallet only ever needs enough to post dispute and finalize transactions.
  • D5.4 Channel acceptance policy. v1 auto-accepts up to 100 USDC per counterparty and limits the hub to 10 channels total. This is a throttle, not a ceiling for v2.

How it works

The interesting code path is the routed payment. Alice's CLI is connected to the hub over a WebSocket. Bob's CLI is connected over a different WebSocket — long-lived, because Bob is running tainnel listen. Both have an open channel with the hub. Alice tells the hub "pay 0.05 USDC to Bob, here is an HTLC locked under hash H, expiring at T". Several things must happen, in this order, all of them inside the hub:

The router checks that both channels (Alice→Hub and Hub→Bob) are open and healthy in the channel-pool. It computes the outbound amount minus the hub's fee using fee-policy. It calls liquidity.reserveOutbound on the Hub→Bob channel, which atomically subtracts capacity so two simultaneous payments cannot both promise the same coin. It then sets the outbound HTLC's expiry to the inbound expiry minus a safety buffer — this is the expiry-buffer arithmetic that makes router.ts the most fragile file in the codebase. Too small a buffer and the hub risks paying downstream while losing the upstream race. Too large and many otherwise valid payments are rejected.

The hub forwards the HTLC to Bob, waits for Bob's preimage, and on success propagates the preimage back to Alice. Both channels then progress one version forward. If Bob never responds, the outbound HTLC eventually expires; the hub releases the reservation via liquidity.releaseReservation and reports failure to Alice, whose own HTLC is also allowed to expire.

Meanwhile, chain-watcher is running its own loop, calling viem.watchContractEvent against PaymentChannel.sol for the four event types the contract emits. When a ChannelClosingUnilateral arrives, the dispute handler is invoked: it compares the version posted on chain to the version it has in SQLite, and if its own state is fresher it builds a dispute() transaction, signs it with the hub's hot wallet, and submits it before the dispute window closes.

Wire protocol vs. agent surface. The hub speaks the v1 wire protocol. Agents in v1 do not implement that protocol directly — they shell out to tainnel pay, which uses the SDK, which uses this WebSocket. Non-TS agents reach the hub by going through the CLI, not by reimplementing the protocol.

Verification model

The hub needs both small unit tests and whole-flow integration tests because its interesting behavior is coordination: a valid payment touches channel state, routing, liquidity reservation, WebSocket messages, database persistence, and chain-event response. Unit tests are useful for deterministic fee policy, route math, nonce replay checks, and typed failure modes. Integration scenarios should run against an anvil fork so channel registration, end-to-end payment routing, stale-state observation, and dispute submission all exercise the same message shapes used by the SDK and CLI.

Design constraints

The hub is an LSP-like counterparty, not a trusted custodian. It can route quickly because it holds channels with users, but it must never be able to create money, ignore signed state ordering, or escape the dispute rules enforced by the contracts.

  • Durable latest state: every signed state the hub might rely on in a dispute must be persisted before the hub acknowledges the transition.
  • Per-channel serialization: concurrent updates for the same channel must pass through one ordered lane so versions cannot fork locally.
  • Expiry buffer: the upstream HTLC expiry must exceed the downstream expiry by enough time for the hub to settle or fail safely.
  • Idempotent payment handling: retried messages should return the same outcome instead of creating duplicate HTLCs.
  • Signed request auth: WebSocket messages are authenticated by signatures and nonce replay protection, not by long-lived bearer tokens.
  • Chain watcher as truth source: on-chain close/dispute/finalize events override stale local assumptions.
