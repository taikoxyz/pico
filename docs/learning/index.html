pico learning

A component-by-component tour of a 1-hop state channel network for AI agents on Taiko L2.

pico is a payments system designed for AI agents to pay each other for work — API calls, model inference, data, anything that can be priced per request. It uses state channels: a 2-of-2 escrow contract on Taiko, with all the actual payments happening off-chain via signed messages. On-chain you only ever see the channel opening, the channel closing, and (rarely) a dispute.

This folder is the friendly tour: open 00-big-picture.html first if you want intuition before structure, or click through the components in order if you'd rather build the picture from the bottom up. Every page is a single self-contained file — no JavaScript, no internet required.

Glossary

Linked from every component page. Skim these once and the rest of the tour reads cleanly.

TermMeaning
channel A 2-of-2 escrow on-chain between two parties (typically agent + hub). Funds are deposited once on open; balances move off-chain via signed updates.
state A snapshot of a channel: (balanceA, balanceB, htlcs[], version), signed by both parties. Versions only go up.
cooperative close Both parties sign a final state and settle in one tx. The happy path.
unilateral close One party (e.g., the hub disappeared) posts the latest state they have on-chain. Opens a dispute window.
dispute window A fixed delay (24h in v1) before a unilaterally-posted state becomes final, giving the counterparty time to challenge with a newer signed state.
HTLC Hash-Time-Locked Contract — a conditional payment: "$X is locked. Whoever reveals the preimage of hash H before time T claims it; otherwise it refunds."
watchtower A bot that watches the chain for fraudulent old-state submissions and posts a counter-state before the dispute window closes.
EIP-712 Ethereum's typed-data signing standard. Channel states are signed as EIP-712 typed data so the digest a contract verifies matches what was signed off-chain.
1-hop routing Payments flow agent → hub → recipient via two paired HTLCs. No multi-hop, no graph gossip.
hub A long-running operator that holds channels with many users and routes payments. Analogous to a Lightning LSP.
hot wallet A private key on a server. Convenient, capped at low balance.
Signer interface The SDK abstraction over key custody. v1 backend = encrypted hot key file. Future backends: KMS, Turnkey, Nitro Enclave, EIP-7702 delegation.
pico listen A long-running CLI subcommand for receivers. Holds a WebSocket session to the hub, accepts inbound HTLCs, reveals preimages.
encrypted hot key file The on-disk key store the v1 Signer reads. Passphrase-derived (scrypt) key, sealed with XSalsa20-Poly1305. Permissions 0600.
litestream A tool that continuously streams a SQLite DB to S3-compatible storage for backup and point-in-time restore.

What v1 deliberately does not include

For honesty: the dogfood scope is narrow on purpose. The wallet UI for humans, ERC-8004 agent-identity registration, EIP-7702 / 4337 delegation, TEE / KMS Signer backends, multi-hop routing, ETH (non-USDC) support, and the DVM / Nostr discovery layer are all Phase 2. The project explicitly does not associate with MCP or x402; if those ecosystems want to consume pico, an external adapter can wrap the CLI.

If you came here to learn before working on the codebase, follow the reading order. If you came here because you'll be running the system, jump to 07-agent-runtime.html and treat the rest as reference.
