End-to-end

Whole-stack scenarios. The proof everything fits together.

What is it

The e2e/ folder contains tests that drive the entire system at once: contracts running on an anvil fork of Taiko mainnet, a real apps/hub instance and an apps/watchtower instance running in-process, the SDK and the CLI talking to them, and assertions checking on-chain state alongside off-chain state. Where P3 through P7 verify each surface in isolation, P8 is the only place where they're verified together. It's also where the human security review happens — the sit-down-with-coffee read-through of the most safety-critical files.

Why it exists

Channel-state systems fail in the seams. Each component can be perfectly correct in its own tests and still produce silent fund loss when wired together — for example, when the SDK persists a state at version N+1 but the hub's router expected N for an in-flight HTLC, or when the watchtower observes a stale-state submission but its evidence is signed under a different chain id. The way to catch those bugs is to drive the whole stack against a real EVM (anvil forking Taiko mainnet) with deterministic fixtures and time-warps, then assert that every component ends with the same state.

Key concepts

  • Anvil fork — a local EVM that mirrors Taiko mainnet state at a pinned block. Lets the tests deploy real contracts and time-warp without touching the real chain.
  • Test harness — e2e/src/harness.ts: spins up anvil, deploys contracts via forge script, starts in-process hub and watchtower, hands handles to the test, tears everything down on afterAll.
  • Time-warp — using anvil's evm_increaseTime RPC to advance past the 24-hour dispute window in seconds. Without it, dispute scenarios would be untestable.
  • Internal review — distinct from the e2e tests: a checklist of files a human reads top-to-bottom looking for bugs the tests can't surface. No external audit in v1 (D8.1); no bug bounty (D8.2).

How it works

A typical scenario starts the harness, opens channels via the SDK, drives a payment, and asserts both off-chain state and on-chain state. The dispute scenarios additionally simulate fraud — the test deliberately tells the hub to post an old signed state — and then assert the watchtower observed it, submitted a counter-state with the latest version, and the contract correctly slashed the cheater after the dispute window. Time is controlled via anvil's RPC, so a 24-hour window passes in milliseconds.

Internal security review checklist

The manual review focuses on trust boundaries and ordering rules that automated scenarios can miss:

  • PaymentChannel.sol + Adjudicator.sol read line by line. Reentrancy guards, integer arithmetic, block.timestamp tolerance, signature-domain coverage.
  • state-machine channel.ts, htlc.ts, signing.ts. Confirm htlcsRoot matches the contract byte-for-byte (the oracle fixture from P3 already enforces this; you re-read for sanity).
  • router.ts: expiry-buffer math (upstream HTLC expiry must be strictly greater than downstream by at least EXPIRY_BUFFER_SECONDS).
  • dispute-handler.ts: always submits the latest known state; bounded gas-bump retries; logs loudly if local state is older than the disputed state.
  • responder.ts: same dispute logic but cold-key; never tries to sign a state it hasn't observed.
  • SDK client.ts pay(): persist-before-send is real, not a comment.
  • CLI cmd/listen.ts: the persist-before-ack ordering.
  • CLI signer/hot-key-file.ts: file permissions 0600, scrypt parameters meet spec, wrong-passphrase returns typed error, raw private key wiped after signing.

Optional automated pass: slither . on the contracts; triage findings.

Verification model

The e2e suite is the place where local correctness becomes system correctness. It should run contracts on an anvil fork, start the hub and watchtower in process, drive the CLI/SDK as real clients, and assert both on-chain and off-chain state after each scenario. Contract invariant fuzzing belongs beside these scenarios as a funds-safety check: it shakes the on-chain state machine while the e2e tests prove the components agree about normal and adversarial flows.

Design constraints

  • Forked chain, not mocks: the harness uses an anvil fork so contract calls, events, timestamps, and finalization behave like an EVM.
  • Real process boundaries: the CLI, hub, watchtower, SDK, and contracts communicate over their actual public surfaces.
  • Deterministic fixtures: keys, contract addresses, block numbers, and time warps must be reproducible so failures can be replayed.
  • Adversarial scenarios: hub-down, receiver-offline, replay, unilateral close, and stale-state dispute flows are as important as the happy path.
  • Human review remains separate: tests find executable regressions; line-by-line review looks for trust-boundary mistakes the scenarios may not cover.
