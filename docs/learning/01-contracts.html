The on-chain contracts

The trustless core. Solidity 0.8.26 on Taiko mainnet.

What it is

The packages/contracts/ directory holds every line of Solidity that pico runs on-chain. It is small on purpose: two real contracts plus a small library, deployed behind UUPS proxies on Taiko mainnet (chainId 167000). Together they are the only thing standing between an honest user and an evil counterparty. Every off-chain signature, every state update, every HTLC eventually resolves back to a check these contracts perform — or it never gets enforced at all. If you only read one package in the repo, read this one.

Why it exists

State channels move payments off-chain by replacing on-chain transactions with co-signed messages. That is cheap and fast, but only if there is a credible threat that an off-chain message will be enforceable on-chain when someone refuses to cooperate. The contracts are that threat. They custody the deposited USDC, they verify EIP-712 signatures, they run a 24-hour dispute window during which a fresher state can override a stale one, and they implement the slashing rule that makes cheating strictly worse than honesty. Without this layer, the entire off-chain protocol would be a polite suggestion.

Key concepts

Skim these from the glossary first if anything looks unfamiliar: channel, state, cooperative close, unilateral close, dispute window, HTLC, watchtower, EIP-712. Two more to keep in mind for this page:

  • UUPS proxy — Universal Upgradeable Proxy Standard. The address you call is a thin proxy; the implementation lives at a separate address and can be swapped via _authorizeUpgrade (gated by onlyOwner). It exists so a bug found in v1 can be patched without forcing every user to migrate to a new channel address.
  • Adjudicator — a pure verifier contract. It holds no funds, has no state to mutate, and exposes only typed-data hashing and signature recovery helpers. Splitting it out of PaymentChannel means we can audit signature logic in isolation and reuse it from off-chain code that wants exactly the same digests the chain will check.

How it works

A channel's life is described by a single status enum on its Channel record. Every external function on PaymentChannel.sol moves that enum forward, and never backward. Funds only leave the contract when status reaches Closed.

The cooperative path is one transaction: both parties sign a final CooperativeClose struct, anyone submits it, the adjudicator verifies both signatures over EIP-712 typed data, and the contract pays out and marks the channel Closed. The unilateral path is the interesting one. closeUnilateral requires only the counterparty's signature on the latest state — the closer's signature is implicit because they are the msg.sender. Submitting moves status to ClosingUnilateral and arms a 24-hour disputeDeadline. Anyone with a fresher co-signed state can call dispute within that window to bump postedVersion. If the closer was actually cheating, a watchtower (or anyone holding a signed update from the cheater) calls submitPenaltyProof, which sets penalized=true. After the deadline anyone calls finalize: if penalized, the entire pot — both sides' deposits — pays out to the honest counterparty. The cheater forfeits their stake.

v2 adds trustless on-chain HTLC settlement. When a unilateral close posts a state with htlcsCount > 0, the dispute window proceeds as usual, then finalize transitions the channel into a new Status.ResolvingHtlcs phase. During that phase, anyone can call claimHtlc(channelId, htlc, proof, sortedIndex, totalLeaves, preimage) — the contract verifies sha256(preimage) == paymentHash and checks an ordered Merkle proof against the posted htlcsRoot, then credits the receiver side. After expiry, refundHtlc(...) follows the same proof shape and credits the sender side. The conservation invariant balanceA + balanceB + htlcsTotalLocked == amountA + amountB is signed into ChannelState and enforced at every close/dispute/topUp entry point, so a state with bogus HTLC totals can't be posted. Watchtowers persist full HTLC sets per signed state so they can build proofs at resolution time.

Design constraints

The contracts are upgradeable UUPS proxies, so the owner key is part of the trust boundary: it can authorize implementation upgrades, but it cannot bypass the state-channel accounting rules inside a finalized implementation. Operational custody of that owner key is tracked outside the learning material.

  • UUPS upgradeability keeps fixes possible without migrating every channel, but it makes owner-key custody security-critical.
  • Reentrancy guards protect token-transfer paths, and state transitions are written so external calls happen after accounting decisions.
  • EIP-712 domain pinning binds signatures to the Taiko chain id and verifying contract, preventing cross-chain or cross-contract replay.
  • Deterministic channelId derivation lets off-chain code and chain events agree on the same channel identity.
  • 100% slashing on proven stale state makes cheating expensive enough for the low-value v1 trust model.
