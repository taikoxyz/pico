1 · Throughput, Latency, and Fees

Throughput

Pico's throughput is bounded by hub capacity and the number of concurrent open channels, not by any protocol limit. Off-chain state updates require only one round-trip between client and hub, so individual channel throughput is CPU- and network-limited in practice. No TPS figure is published. For context:

  • Lightning processes ~8 million transactions per month network-wide (early 2025); each channel is independent, no shared bottleneck.
  • Visa/Mastercard handles 1,700 avg / 65,000+ peak TPS on a centralised shared ledger — not meaningfully comparable to channel throughput.
  • XRP Ledger sustains ~1,500 on-ledger TPS, theoretical ceiling 65,000+.
  • Raiden produced 300–500 TPS in simulated lab conditions; no live network numbers exist.
  • Connext moved away from state channels and is now an intent-based cross-chain bridge targeting <60 s latency — not a micropayment channel by architecture.

Latency

Pico's payment latency — one round-trip off-chain, sub-second — is a clear structural advantage over every on-ledger system. There is no mining, no block confirmation, no routing graph traversal. Lightning is comparable in the happy path (sub-second within an open channel) but adds routing retries under congestion and CLTV timeout overhead for multi-hop paths. XRP Ledger's 3–5 second on-ledger finality is too slow for streaming API metering loops. Visa authorization takes seconds but clearing takes 1–3 days; authorization is not final settlement.

2 · Trust and Decentralisation

Pico's trust surface has three semi-trusted elements: the hub, the watchtower(s), and Taiko L2 itself.

  • Hub: cannot steal funds (HTLCs are hash-locked), but can refuse service or stall a cooperative close. User's recourse is unilateral close plus a 24-hour dispute window — structurally identical to how Lightning routing nodes behave.
  • Watchtower: cannot steal; can fail to post a penalty proof within the 24-hour window. Recommended deployment is 2-of-3 independent operators.
  • v1 HTLC limitation (material): unilateral close rejects any state with in-flight HTLCs — they revert to an empty root on-chain. This is a trusted-liveness assumption, not fully trustless. On-chain HTLC settlement is deferred to v2. This is pico's most significant trust gap versus Lightning.

Pico and Lightning are comparable in trust design. Pico's 24-hour dispute window is roughly equivalent to Lightning's worst-case stacked CLTV delta (~144 blocks ≈ 24 hours at Bitcoin's 10-min block time), making exit latency parity between the two.

Visa/Mastercard's centralisation risk is categorically different: either network can de-platform a merchant unilaterally, freeze funds during disputes, and mandates KYC/AML for all participants. XRP is decentralised for native XRP transfers but Ripple's UNL governance creates concentration risk; the August 2025 SEC settlement ($50 M cash) resolved the immediate legal overhang but established that institutional XRP distribution can trigger securities liability.

3 · Liquidity and UX

Channel setup

Opening a pico channel requires one on-chain tx on Taiko L2 (gas is cheap). The minimum channel is 10 USDC. The hub then tops up inbound liquidity via a second on-chain tx. Compare to Lightning's JIT channels: LSPs like Phoenix open a channel automatically on first receive, with the liquidity fee bundled into the payment — no explicit two-step ceremony. Phoenix charges 0.4% for JIT inbound channel creation, which is 4× pico's ongoing payment fee but eliminates manual channel management.

Liquidity efficiency

Pico's hub auto-recycle (the RECOMMENDED behavior in §8.8 of the protocol spec): recovered USDC from cooperative closes is immediately recycled to fund the next channel's topUp. A hub can serve N simultaneous channels with less than N × channel_size in capital because not all channels reach capacity simultaneously. Lightning has the same structural locked-capital problem but with more mature rebalancing tooling (circular rebalancing, submarine swaps, splicing). In practice, Lightning channel liquidity is highly imbalanced: the top 10% of hubs control 97% of liquidity (Gini ~0.97), and routing success collapses below 1% for payments above $200.

Agent integration UX

Pico's CLI (pico pay --to <addr> --amount <n> --json) is explicitly designed for headless agent use. Non-TypeScript callers parse the --json output. No wallet, no browser, no signing UI required.

  • Lightning: BOLT-compatible REST APIs exist but require node operator knowledge. No purpose-built agent CLI.
  • Connext / Raiden: Ethereum-ecosystem tooling requiring wallet integration.
  • Visa: Mastercard Commercial Connect offers B2B APIs for enterprise automation, not API metering between agents.
  • XRP: Payment channels exist on-ledger but are essentially undocumented for agentic use cases.

5 · Competitive Positioning

Where pico is stronger

  1. AI-agent-first design. No other system ships a purpose-built headless CLI for agent payments with --json output. This is a real moat in v1's target market.
  2. USDC-native, no protocol token. Stable denomination eliminates the BTC volatility tax on Lightning. No token to manage, stake, or speculate on — removes an entire category of user friction and governance attack surface.
  3. Simplest viable topology. 1-hop eliminates routing failures. Lightning's <1% routing success rate on payments above $200 is a direct consequence of multi-hop complexity. Pico trades routing flexibility for deterministic delivery.
  4. Fee efficiency for USDC. 0.10% beats Visa (2–3%) and Connext (<0.20%), and is competitive with Lightning (0–0.1%) while offering stablecoin settlement.
  5. No governance, no bridges. Eliminates entire classes of attack surface (governance attacks, bridge exploits). The system cannot be upgraded without owner key action; there are no token holders to vote.
  6. Capital recycling. Hub auto-recycle (§8.8) allows one pool of USDC to serve multiple sequential channels without idle periods. No equivalent is documented for Raiden or Connext.

Where pico is weaker

  1. v1 HTLC trust gap. In-flight HTLCs revert to empty root on unilateral close. Lightning's on-chain HTLC settlement is fully trustless. This is the most material gap for high-value or long-duration conditional payments. Scheduled for v2.
  2. Single-asset (v1). USDC only limits interoperability. Lightning is increasingly multi-asset via plugins. XRP Ledger supports any issued token. This is a design choice, not a limitation, but it restricts addressable market.
  3. Hub concentration risk. Every payment touches one specific hub. A hub going offline is service disruption — worse than a Lightning routing failure, where the wallet can retry a different path. Multi-hub support mitigates but doesn't eliminate this.
  4. Taiko-only. Not portable to other EVM chains. Lightning (Bitcoin) and Connext span many ecosystems. Taiko's footprint vs. Solana/Base is pico's biggest external market risk.
  5. No privacy. On-chain channel IDs are deterministic; the hub sees all amounts and payment hashes. Lightning's onion routing, while weak in practice due to topology analysis, provides more than pico's zero privacy.
  6. Unproven scale. All throughput/latency claims are architectural, not empirically measured. Lightning's 8 M+ monthly transactions is a real data point; pico has none yet.
  7. Watchtower dependency. The 24-hour dispute window with no live watchtower is unmitigated. Lightning has the same structural dependency but a more mature watchtower ecosystem and wider tooling.

Verdict

The case for pico

Pico occupies a defensible niche that no existing system targets well: USDC-denominated, sub-cent, headless agent-to-agent payments on a low-cost EVM L2 with no token or governance overhead. It beats Lightning on stablecoin settlement and UX simplicity; it beats Visa/Mastercard and Ripple on micropayment economics by 10–30×; it beats Raiden and Connext on production readiness and design clarity.

For its stated target — AI agents calling APIs, paying DVMs, metering sub-cent API usage — the 1-hop design is well-justified. Routing failures (Lightning's Achilles' heel above $200) simply don't exist in a topology where every payment takes exactly one hop through a known hub.

The risks

Protocol: The v1 trusted-liveness HTLC assumption means pico is not fully trustless for in-flight payments. Lightning remains more trustless in this specific scenario. Until v2 ships on-chain HTLC settlement, this is an honest limitation that must be disclosed to users.

Market: x402 (Coinbase/Base), Solana Pay.sh, and Amazon Bedrock Agentcore Payments are all emerging in the same "agent payments" narrative on higher-adoption chains. Taiko's current footprint vs. Solana and Base is pico's biggest external risk — not its protocol design. The protocol is sound; the ecosystem bet on Taiko is the open question.

vs. Lightning

Comparable trust model. Pico wins on stablecoin denomination, UX simplicity, and routing determinism. Lightning wins on raw fee rate, privacy (onion), trustless HTLC settlement, and ecosystem maturity.

vs. Raiden / Connext / Perun

Pico is the only one targeting production with a working product. Raiden has minimal live adoption. Connext abandoned state channels. Perun is research-grade. Pico wins on clarity of purpose and production readiness.

vs. Visa / Mastercard

Not the same market. Visa is 10–30× more expensive for online payments, structurally broken below $2, and requires full KYC. Pico addresses a workload Visa cannot serve economically.

vs. Ripple / XRP

XRP Ledger has faster finality on-ledger and lower per-tx cost, but requires XRP for gas, has institutional rather than agent focus, and its payment channel feature is unused in practice. Pico is better suited to agent micropayments.
