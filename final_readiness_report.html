2.Architecture snapshot

pico is a 1-hop state-channel network: every channel terminates at a hub, every payment is client → hub → recipient. v1 supports a single asset per channel; the contract surface accepts native ETH (token == address(0)) and any owner-allowlisted ERC-20, with USDC as the v1 default. The protocol is frozen at version: '2' in the EIP-712 domain.

3.3 Findings

SC-01 — External audit gate not satisfiedCRITICAL

docs/audit-status.md:50,76 says: "No v2.1.* tag may be cut until an external auditor (Spearbit / Trail of Bits / Cantina) provides written sign-off." v2.1.9 is tagged. Either the policy must be amended in writing or an external auditor's report must be added to docs/audit/. Pico has been DeepSeek-audited and internally multi-agent-reviewed but neither is equivalent to an external human audit for an on-chain custody surface.

Action: Engage an audit firm; freeze new v2.x features until written sign-off is in repo.

SC-02 — Owner is an EOA (PC-09 still Open)CRITICAL

Until the Taikoscan owner readout shows a TimelockController address with getMinDelay() == 172800 and a Safe multisig as proposer/executor, a single key compromise can change the token allowlist, change min amounts, or push a malicious UUPS upgrade. TransferOwnership.s.sol is correct; it just hasn't been run on mainnet.

Action: Run TransferOwnership.s.sol; record the tx hash, the new owner contract address, and the Safe signer set in docs/launch-log.md. Sweep + cold-store the deployer EOA immediately afterwards.

SC-03 — Foundry scripts do not assert block.chainidHIGH

Operator error could deploy to the wrong network. Mitigation: mainnet-deploy.md runbook line-item + bytecode-parity post-deploy check (already present in e2e/src/scenarios.fork.test.ts for the v2.1.x bytecode).

SC-04 — Fee-on-transfer hardeningMEDIUM

Owner discipline is the only thing preventing a future allowlist of a fee-on-transfer token from breaking balance conservation. Doc-only mitigation today; consider a runtime "check received == requested" gate in v2.2 if scope expands beyond ETH+USDC.

4.Off-chain runtime audit

4.1 Hub (apps/hub)

  • Production gates — config-validate.ts:10–87 rejects all 7 well-known dev keys on non-Anvil chainIds, demands explicit HUB_PRIVATE_KEY, mandates HUB_REQUIRE_SIGNED_ENVELOPE=true, mandates HUB_OPERATOR_TOKEN, refuses zero contract addresses, and refuses chains whose protocol map slot is still ZERO_ADDRESS. Mainnet startup will fail loud rather than fall back to a defaulted dev key.
  • Admit gates — apps/hub/src/api/ws.ts imports admitSignedState, admitHtlcOffer, admitHtlcSettle, admitHtlcFail, admitClose from @inferenceroom/pico-state-machine; each is wired into the corresponding ingestion path before any DB write. verifyEnvelope is enforced (line 892) when requireSignedEnvelope is true.
  • Per-channel mutex — KeyedMutex<ChannelId> in ws.ts:115 guards pay-route construction (incoming+outgoing). Locks are taken in a deterministic order; deadlock-free.
  • Reservation/durability — channelPool.hydrate(), liquidity.hydrate(repos.htlcs), and snapshot seeding from the latest co-signed state run before the chain watcher starts (server.ts:64–88). In-flight routes are persisted in payment_routes; router.loadInflight rehydrates them on restart.
  • Chain watcher — chain-watcher.ts chunks getLogs to ≤500 blocks; rewindForReorg detects block-hash mismatch and falls back one chunk. Caveat: chunk-based rewind is correct only for reorg depths < chunk size. Taiko mainnet historical reorgs are well below 50 blocks; assumption documented but not asserted.
  • Dispute handler — dispute-handler.ts picks latestDisputeEligible(), compares against the attacker's posted version, refuses sentinel bootstrap v0 states.
  • TopUp + auto-recycle — topup-handler.ts enforces one in-flight topup per channel; auto-recycle.ts hydrates and re-uses recovered liquidity in the same token. Hot-wallet mutex serializes liquidity operations.
  • Metrics binding — server.ts:223–229 binds PROMETHEUS_PORT to METRICS_BIND_ADDR (defaults to 127.0.0.1 in production). In the GKE manifest the bind address is widened to :: for in-cluster scrape (infra/k8s/01-hub.yaml:27); ingress is locked down by NetworkPolicy.

4.2 Watchtower (apps/watchtower)

  • Production gates — config-validate.ts rejects dev keys, requires explicit WATCHTOWER_PRIVATE_KEY, refuses MODE=service, validates penalty threshold + scheduler interval + confirmations.
  • R-04 wallet nonce race — mutex.ts + responder.ts wrap _submitPenalty in walletMutex.run(). responder.concurrentNonce.test.ts asserts distinct nonces under concurrent submission.
  • R-05 reorg / inclusion bookkeeping — Observations table carries block_hash and block_number; rewindForReorg(fromBlock) clears inclusion state for affected channelIds. Covered by responder.reorg.test.ts.
  • WTW-001 confirmation flush — watcher.ts has an independent confirmationFlushInterval that flushes pending events that crossed confirmation depth even with no new logs.
  • WTW-003 stuck-tx replacement — responder.ts:121–150 replaces stuck tx with the same nonce and bumped fee after inclusionTimeoutMs.
  • WTW-010 receipt status — responder.ts:144,237 require receipt.status === 'success'.
  • HTLC resolver — htlc-resolver.ts issues claimHtlc with Merkle proof when a preimage is known and HTLC not expired, otherwise refundHtlc after expiry. Idempotent because the contract refuses double-settlement.

4.3 SDK + CLI + state-machine

  • F-01 admit gates — SDK client imports admitSignedState at lines 20–23 and calls it before persisting any hub-supplied state (paths at 389, 627, 785, 916). The state-machine is the single point of truth.
  • F-03 invoice replay — storage-file.ts:170 markInvoiceConsumed is idempotent; client.ts:419–421 rejects already-consumed invoices.
  • F-05 hot-key durability — storage-file.ts:1,52–55,60,79,104–108 fsyncs the file and parent dir; 0o600 files and 0o700 directories. Encryption: scrypt(N=2^17, r=8, p=1) + xsalsa20-poly1305.
  • EOD-01 verifyingContract — CLI and SDK use CONTRACT_ADDRESSES[chainId].Adjudicator as the EIP-712 verifyingContract, matching the contracts' own domain. Hub uses the same.
  • State-machine admit — admit.ts exposes admitSignedState, admitHtlcOffer, admitHtlcSettle, admitHtlcFail, admitClose. 138/138 tests pass.

4.4 Findings

RT-01 — Operator-token compare is not constant-timeHIGH

Hub's bearer-token check uses string === (timing-sensitive). For an over-the-internet admin token with 32 bytes of entropy this is a small risk, but it is cheap to fix. Action: replace with crypto.timingSafeEqual + length check.

RT-02 — Hub /v1/health probe is shallowMEDIUM

Probes DB ping and RPC getBlockNumber, but not chain-watcher staleness, pending-tx age, or scheduler tick. Operators can't see "hub is running but events aren't being processed." Action: expose chainWatcherLagBlocks and oldestPendingTxAgeMs in health JSON; consider returning 503 above documented thresholds.

RT-03 — Reorg rewind is chunk-granular, not per-block-hash walkMEDIUM

Safe today on Taiko (shallow reorgs), but the assumption should be explicit. Action: document the depth assumption in docs/threat-model.md and add a metric alert if observed rewind depth approaches chunk size.

RT-04 — HSTS header not set in hub appMEDIUM

infra/k8s/01-hub.yaml:148–152 already calls this out: GCE Ingress does not inject HSTS, so the hub must. Action: 5-line Fastify middleware emitting Strict-Transport-Security: max-age=63072000; includeSubDomains; preload.

RT-05 — H-10 / H-11 still "Patched-not-reaudited"MEDIUM

SDK defaults to DEFAULT_HUB_FEE_BPS/FLAT from protocol constants rather than honoring a hub-advertised fee policy. Liquidity is hydrated from the latest co-signed state plus snapshot; full authoritative liquidity-from-states is a v2.2 item. Both should be disclosed in the v1.0 release notes.

RT-06 — Graceful shutdown does not explicitly drain WSLOW

server.ts:279–284 stops the chain watcher and closes the DB but does not drain active WS sessions. Acceptable for v1; document client reconnect window in SDK README.
